# Chapter 4: The WorkRequest Protocol

## A Work Order, Not a Function Call

In a traditional microservice architecture, services communicate via API calls: HTTP
requests with typed JSON payloads, versioned endpoints, and schema contracts. When the
schema changes, clients break. This is the tyranny of typed contracts in distributed
systems.

In a hotel, work orders do not look like API calls. When room service tells the kitchen
"Club sandwich, room 403, no tomato, add extra bacon," nobody checks a JSON schema. The
kitchen understands the request because the language is clear and the context is shared. If
room service says "the usual for the guest in 403" and the kitchen knows what that means
from past experience, that works too.

The Ensemble Network's communication primitive is the **WorkRequest**: a standardized
envelope that carries work between ensembles. It is not a function call. It is a work
order. The content is natural language. The envelope provides the operational metadata:
who is asking, what priority, when it is needed, and how to deliver the result.

## The Envelope

```java
public record WorkRequest(
    String requestId,
    String from,
    String task,
    String context,
    Priority priority,
    Duration deadline,
    DeliverySpec delivery,
    String traceContext,
    CachePolicy cachePolicy,
    String cacheKey
) {}
```

Each field serves a specific purpose:

**requestId**: The unique identifier for this specific work request. Generated by the
caller. Serves two roles: correlation (matching the response to the request) and
idempotency (if the same request is sent twice due to a retry, the receiver recognizes the
duplicate and returns the cached result rather than executing again).

**from**: The name of the requesting ensemble. Used for routing the response and for audit
trail logging.

**task**: The name of the shared task or tool to execute. Must match a name declared via
`shareTask()` or `shareTool()` on the target ensemble.

**context**: The natural language input for the task. This is the "what to do" -- the
content of the work order. For a NetworkTask, this is the full task context that the target
ensemble's agent will work with. For a NetworkTool, this is the tool input.

**priority**: CRITICAL, HIGH, NORMAL, or LOW. Determines ordering in the target ensemble's
priority queue. CRITICAL work is processed before HIGH, which is processed before NORMAL,
and so on.

**deadline**: The caller's SLA. "I need this within 30 minutes." The target ensemble is
not required to meet the deadline -- it reports its estimated completion time, and the
caller decides whether to wait, cancel, or try elsewhere.

**delivery**: How and where to return the result. A `DeliverySpec` containing a method
(WebSocket, queue, topic, webhook, store, broadcast-claim, or none) and a method-specific
address. This decouples the response path from the request path.

**traceContext**: W3C Trace Context headers (`traceparent` and `tracestate`) for distributed
tracing. Mandatory. Allows the cross-ensemble call to be correlated in Jaeger, Tempo, or
any OpenTelemetry-compatible backend.

**cachePolicy**: USE_CACHED or FORCE_FRESH. When USE_CACHED, the target ensemble may return
a previously computed result for the same `cacheKey` if it is within the `maxAge`. When
FORCE_FRESH, the target always executes.

**cacheKey**: Optional key for result caching. Different from `requestId` (which is for
idempotency). The cache key represents "this type of request" rather than "this specific
request instance."

## "Call Me Back at This Number"

The delivery specification is the most novel part of the WorkRequest. In traditional
request-response systems, the response goes back the way the request came in. An HTTP
response goes back on the same TCP connection. A WebSocket response goes back on the same
WebSocket.

In the Ensemble Network, the response path is explicitly specified by the caller. This
enables a pattern familiar from everyday life: "Call me back at this number when you have
a hotel room available."

The caller says: "Here is my work request. When you have the result, deliver it to
`kafka://maintenance.results`." The target ensemble processes the request, and when the
result is ready, publishes it to the specified Kafka topic. The caller (or any replica of
the caller) consumes it from there.

This decoupling is essential for production deployment. In Kubernetes, pods come and go.
The pod that sent the request might be killed before the response arrives. The pod that
processes the request might be a different replica than the one that accepted it. The
response needs to reach the caller regardless of which pods are currently running. The
delivery specification makes this explicit.

### Delivery Methods

**WEBSOCKET**: Direct, real-time delivery over a WebSocket connection. The simplest method.
Used in development and for fast, synchronous interactions where both parties are connected.

**QUEUE**: Point-to-point delivery via a durable queue (Redis Streams, SQS). The result is
enqueued; exactly one consumer processes it. Survives pod restarts.

**TOPIC**: Pub/sub delivery via a message topic (Kafka). The result is published; all
subscribers receive a copy. Useful when multiple ensembles need the same result, or when
the audit system needs a copy.

**WEBHOOK**: HTTP POST to a specified URL. Useful for integrating with external systems
that are not part of the ensemble network.

**STORE**: Write the result to a shared key-value store (Redis) keyed by the `requestId`.
The caller polls or subscribes for the result. Useful for very large results or when the
caller wants to retrieve the result at its convenience.

**BROADCAST_CLAIM**: Announce the result to all replicas of the requesting service. The
first replica to claim it receives the full payload. Useful when the caller has multiple
replicas and the result should be processed by whichever one is free.

**NONE**: Fire and forget. The caller does not need a response. Useful for notifications
or background processes.

## Ingress: How Work Arrives

The delivery specification controls how results leave an ensemble. The ingress
configuration controls how work arrives.

A long-running ensemble can accept work from multiple sources simultaneously:

**WebSocket**: Direct from another ensemble over a persistent WebSocket connection. The
simplest and fastest ingress. Used when ensembles have direct connections.

**Queue**: Pull from a durable queue. The ensemble polls or subscribes to a queue (Redis
Streams, Kafka, SQS) and picks up work items. Multiple replicas of the same ensemble can
read from the same queue with consumer groups, providing natural load balancing.

**HTTP API**: Accept work via `POST /api/work`. Useful for external systems, scripts, CI
pipelines, or any client that can make HTTP requests. The request body is a WorkRequest
JSON.

**Topic subscription**: React to events published on a topic. When the front desk publishes
a "guest checkout" event to `hotel.events`, the housekeeping ensemble (subscribed to that
topic) receives it and triggers a room cleaning task.

**Schedule**: Internal cron or interval. Scheduled tasks generate WorkRequests internally
on a timer, without any external trigger.

All ingress sources normalize their input to the same `WorkRequest` envelope. Once
normalized, the work item enters the ensemble's priority queue and is processed in priority
order regardless of how it arrived.

## The Request-Response Cycle

Putting it all together, a typical cross-ensemble interaction follows this cycle:

```
1. Caller constructs WorkRequest
   (requestId, task, context, priority, deadline, delivery, trace)

2. WorkRequest is sent via the chosen ingress method
   (WebSocket direct, queue, HTTP, topic)

3. Target ensemble receives and acknowledges
   -> task_accepted { requestId, queuePosition, estimatedCompletion }

4. Target ensemble processes the request
   (agent synthesis, ReAct loop, tool calls, review gates)

5. Target ensemble delivers the result
   via the delivery method specified in the WorkRequest

6. Caller receives the result
   (unblocks if AWAIT mode, triggers callback if ASYNC mode)
```

The acknowledgment in step 3 is important. The target ensemble does not just silently
accept the work. It responds immediately with its queue position and estimated completion
time. This gives the caller the information it needs to make routing decisions: wait, cancel
and try elsewhere, or continue without the result.

## Why Natural Language Context?

The `context` field in a WorkRequest is natural language, not a typed schema. This is a
deliberate design decision that eliminates an entire category of distributed systems
problems.

In a traditional microservice, changing a field name in a request payload breaks all
callers. Adding a required field breaks all callers. Removing a field breaks all callers.
Schema versioning, backward compatibility, and API evolution consume enormous engineering
effort.

In the Ensemble Network, the context is: "Order replacement valve model X-420 for building
2 boiler, urgent priority." If the procurement ensemble changes how it parses this -- maybe
it now extracts structured data more intelligently, or handles a wider variety of request
formats -- the caller is unaffected. The caller sent natural language. The receiver
interpreted it.

The LLM on the receiving end is the **compatibility layer**. It interprets the natural
language context regardless of exact wording, phrasing, or format. Minor changes do not
break callers. This is not a weakness of the protocol -- it is a strength. It trades the
precision of typed schemas for the resilience of natural language interpretation, which is
the right trade for LLM-based systems.

When structured data is needed for specific fields (like a purchase order number or a
delivery date), the structured output mechanism (`outputType`) handles it. But the task
description and the context -- the "what to do" -- remains natural language.
